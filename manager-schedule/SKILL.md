## Foundation


### Overview
manager-schedule/SKILL.md는 사용자의 루틴, 일정, 할 일, 이벤트를 skills/registry-schedule/SKILL.md에 기록하여 관리하는 SKILL이다.


### Triggers
- registry-schedule/SKILL.md의 Last Updated 날짜와 현재 날짜가 다른 상황
- registry-schedule/SKILL.md의 업데이트가 필요한 상황
- registry-schedule/SKILL.md에 문제가 있는 상황
- registry-schedule/SKILL.md에 항목 추가 혹은 제거가 필요한 상황
- 사용자가 registry-schedule/SKILL.md의 수정을 요청하는 상황


### Usage
1. 현재 날짜 및 시각을 확인한다.
2. skills/registry-schedule/SKILL.md를 읽는다.
3. 업데이트 사항을 확인한다.
4. 면밀히 검토한다.
5. skills/registry-schedule/SKILL.md를 업데이트한다.


## Context


### Purpose
사용자의 루틴, 일정, 할 일, 이벤트를 skills/registry-schedule/SKILL.md에 기록하여 관리한다.


### Glossary
- 루틴: 수업, 운동, 과외 등 반복되는 블록
- 일정: 데이트, 저녁 약속, 비즈니스 미팅 등 일회성 블록
- 할 일: 과제, 업무, 신청 등 마감 기한이 있는 작업
- 이벤트: 기념일, 생일, 시험 등 인지하고 있어야 할 날
- D-Day: 현재 날짜를 기준으로 한 잔여 일수 표기 체계. 당일은 D-Day, 미래는 D-{남은 일 수}


### Philosophy
- 좋은 것엔 견고함이 있다.
- 그 어떤 표현도 다른 표현으로 대체할 수 없다.
- 공들임은 값지다.
- 좋은 물음엔 맥락이 흐른다.
- 절제는 깊이를 만든다.
- 차분함이 정확함을 만든다.
- 적당한 것은 적절하지 않다.
- 정확하기에 아름답다.
- 일관성이 신뢰를 만든다.
- 그냥은 없다.

### Format Rules
- 섹션 순서: 루틴 → 일정 → 할 일 → 이벤트
- 루틴 정렬: 시작 시각 오름차순. 시각이 같으면 요일순(월→일). 루틴은 첫 번째 요일을 기준으로 요일순 계산.
- 일정 정렬: 날짜 오름차순. 날짜가 같으면 시작 시각 오름차순.
- 할 일 정렬: 마감일 오름차순. 마감일이 같으면 마감 시각 오름차순. 마감 시각이 없으면 해당 날짜 맨 아래.
- 이벤트 정렬: 날짜 오름차순.
- 헤더 시간 표기: 24시간제 HH:mm.
- D-Day 표기: 현재 날짜 기준 당일은 D-Day, 미래는 D-{남은 일 수}.
- 마감 시각이 없는 경우: {YYYY}년 {MM}월 {DD}일 {EEEE(ko)}까지만 작성
- 마감 시각이 정각인 경우: {YYYY}년 {MM}월 {DD}일 {EEEE(ko)} {A(ko)} {hh}시까지만 작성
- 위치: 물리적 위치, 도로명주소, URL, 이메일, 연락처, 계좌번호, 파일 경로 등 맥락에 맞는 접근 정보 작성. 없으면 ‘없음’ 작성. 아직 정해지지 않았으면 ‘미정’ 작성
- 같은 루틴이라도 요일에 따라 시간 또는 위치가 다르면 별개 항목으로 분리
- 섹션(##) 위로 빈 줄 2개. 항목(###) 위로 빈 줄 2개. 그 외 빈 줄 없음.
- 비고: 맥락에 따라 유연하게 작성. 없으면 ‘없음’ 작성.
- 반복: 매주, 격주, 매일, 매달 등의 표현을 활용하여 작성
- 불참: 루틴이 아닌 일정의 비고에 작성

## Rules


### Principles
- D-Day는 Last Updated와 관계없이 무조건 현재 날짜를 기준으로 계산한다.
- 한 항목은 하나의 섹션에만 속할 수 있다.
- registry-schedule/SKILL.md는 유일한 원본이다. 확인, 조회, 수정은 반드시 해당 문서를 기준으로 한다.
- 읽기가 쓰기보다 먼저다. 기록하기 전에 반드시 registry-schedule/SKILL.md의 현재 상태를 확인한다.
- 기록 전 확인은 생략할 수 없다. 기록 전에 반드시 사용자 확인을 거친다.
- 구조가 데이터를 보호한다. 루틴, 일정, 할 일, 이벤트의 분리와 헤딩 계층은 임의로 변경하지 않는다.
- 필드는 삭제하지 않는다. 값이 없어도 필드를 유지한다.
- 정렬은 기록의 일부다. 새 항목을 추가할 때 올바른 위치에 삽입한다.
- 부족한 정보는 추측하지 않고 물어본다. 정보가 불확실하면 사용자에게 확인을 요청한다.
- 보고는 기록 및 관리의 마무리 단계다. 업데이트를 완료한 뒤 변경 내용을 사용자에게 반드시 안내한다.


### Pitfalls
- 마감 시각이 없는 할 일에 시각 필드를 임의로 채우는 실수. 시각이 없으면 날짜까지만 작성한다.
- 마감 시각이 정각인데 분까지 표기하는 실수. 정각이면 ‘오후 3시’처럼 시까지만 작성한다.
- 항목 삭제 후 빈 줄이 3개 이상 연속되는 실수. 섹션 헤더 위 2개, 항목 헤더 위 2개, 그 외 0개를 지킨다.
- 섹션 내 항목이 전부 제거되었을 때 섹션 헤더까지 삭제하는 실수. 빈 섹션이어도 섹션 헤더는 남긴다.
- 날짜가 지났음에도 항목을 제거하지 않고 방치하는 실수. 현재 날짜 이전의 일정·할 일·이벤트는 제거 대상이다.
- 현재 날짜로 갱신되지 않은 Last Updated 날짜를 D-Day 계산에 사용하는 실수. D-Day는 현재 날짜를 기준으로 계산한다.
- 루틴을 요일순이 아닌 가나다순 등으로 정렬하는 실수. 시작 시각 오름차순이 우선이고, 시각이 같으면 요일순(월→일)이다.​​​​​​​​​​​​​​​​
- 일회성 블록을 루틴에 넣거나, 반복되는 블록을 일정에 넣는 실수. 반복 여부가 루틴과 일정을 구분한다.
- 위치가 아직 정해지지 않은 항목에 ‘없음’을 쓰는 실수. 미확정은 ‘미정’, 접근 정보가 존재하지 않는 것이 ‘없음’이다.
- 불참 정보를 루틴의 비고에 쓰는 실수. 불참은 해당 일정의 비고에 작성한다.
- 마감 시각이 없는 할 일을 같은 날짜의 다른 항목 위에 두는 실수. 시각이 없으면 해당 날짜 맨 아래다.
- Last Updated를 갱신하면서 D-Day 값을 재계산하지 않는 실수. 날짜가 바뀌면 모든 D-Day를 재계산해야 한다.
- 반복 여부를 확인하지 않고 루틴 또는 일정에 분류하는 실수. 반복 여부가 확인되지 않으면 사용자에게 질문한다.
- 할 일이나 이벤트 헤더에 HH:mm을 쓰는 실수. 할 일과 이벤트의 헤더는 D-Day 정보이고, HH:mm은 루틴과 일정에서만 쓴다.
- 일정에 반복 필드를, 루틴에 날짜 필드를 쓰는 실수. 루틴은 반복, 일정은 날짜다.
- D-Day 당일을 D-0으로 표기하는 실수. 당일은 D-Day다.
- Last Updated 날짜를 한국어나 ISO 형식으로 쓰는 실수. MMMM DD, YYYY 영문 형식만 허용한다.
- 일정·할 일·이벤트의 날짜를 영문이나 ISO 형식으로 쓰는 실수. 일정·할 일·이벤트의 날짜는 한국어 형식이고, 영문 형식은 Last Updated에서만 쓴다.
- 할 일의 필드명을 ‘날짜’로 쓰는 실수. 할 일은 ‘마감’이고, ‘날짜’는 일정과 이벤트의 필드명이다.
- 비고나 위치 필드를 빈칸으로 남기는 실수. 값이 없으면 ‘없음’, 미확정이면 ‘미정’을 명시한다.


### Edge Cases
- 자정을 넘기는 블록: 예외 없음 {HH:mm}-{HH:mm} | {블록 이름}으로 헤더 작성. (예시: 22:00-01:00 | 심야 자습)
- 여러 날에 걸치는 일정: 날짜 필드에 시작일과 종료일을 하이픈으로 연결. 헤더 시간은 시작일의 시작 시각부터 종료일의 종료 시각까지. 정렬은 시작일 기준. (예시: 2026년 3월 21일 토요일 - 2026년 3월 22일 일요일)
- 루틴 일시 중단: 항목을 삭제하지 않고 비고에 중단 기간 명시. 중단 기간이 지나면 비고 초기화.
- 항목 충돌: 허용하지 않음. 사용자에게 즉시 알림.
- 루틴 종료일: 비고에 작성. 종료일이 지나면 해당 루틴 삭제
