## Foundation


### Overview
presenter-schedule는 registry-schedule를 기반으로 특정 날짜의 일간 일정을 schedule-YYYY-MM-DD.md 파일로 생성·관리하는 스킬이다. 해당 날짜의 루틴과 일정을 일정표로 통합하고, 할 일과 이벤트를 D-Day와 함께 표시하며, ~/.openclaw/workspace/memory/ 경로에 파일을 저장한다.


### Triggers
- ~/.openclaw/workspace/memory/ 경로에 schedule-YYYY-MM-DD.md(해당하는 날짜) 파일이 없는 상황
- schedule-YYYY-MM-DD.md(해당하는 날짜)의 업데이트가 필요한 상황
- schedule-YYYY-MM-DD.md에 문제가 있는 상황
- schedule-YYYY-MM-DD.md(해당하는 날짜)에 항목 추가 혹은 제거가 필요한 상황
- 사용자가 schedule-YYYY-MM-DD.md(해당하는 날짜)의 수정을 요청하는 상황


### Usage
1. registry-schedule을 읽는다.
2. registry-schedule에 업데이트가 필요한 경우 manager-schedule로 업데이트한다.
3. ~/.openclaw/workspace/memory/ 경로의 schedule-YYYY-MM-DD.md(해당하는 날짜)를 읽는다.
4. 업데이트할 내용을 확인한다.
5. schedule-YYYY-MM-DD.md(해당하는 날짜)를 업데이트한다.


## Context


### Purpose
사용자의 일간 일정을 ~/.openclaw/workspace/memory/ 경로에 schedule-YYYY-MM-DD.md로 기록하여 관리한다.


### Glossary
- 루틴: 수업, 운동, 과외 등 반복되는 블록
- 일정: 데이트, 저녁 약속, 비즈니스 미팅 등 일회성 블록
- 할 일: 과제, 업무, 신청 등 마감 기한이 있는 작업
- 이벤트: 기념일, 생일, 시험 등 인지하고 있어야 할 날
- D-Day: schedule-YYYY-MM-DD.md 날짜를 기준으로 남은 일 수


### Philosophy
- 좋은 것엔 견고함이 있다.
- 그 어떤 표현도 다른 표현으로 대체할 수 없다.
- 공들임은 값지다.
- 좋은 물음엔 맥락이 흐른다.
- 절제는 깊이를 만든다.
- 차분함이 정확함을 만든다.
- 적당한 것은 적절하지 않다.
- 정확하기에 아름답다.
- 일관성이 신뢰를 만든다.
- 그냥은 없다.

### Format Rules
- 섹션 순서: 일정표 → 할 일 목록 → 이벤트
- 일정표 정렬: 시작 시각 오름차순.
- 할 일 목록 정렬: 마감일 오름차순. 마감일이 같으면 마감 시각 오름차순. 마감 시각이 없으면 해당 날짜 맨 아래.
- 이벤트 정렬: 날짜 오름차순.
- D-Day 표기: schedule-YYYY-MM-DD.md 날짜 기준 당일은 D-Day, 미래는 D-{남은 일 수}.
- 헤더 시간 표기: 24시간제 HH:mm.
- 할 일 목록 포함 범위: schedule-YYYY-MM-DD.md 날짜 기준 마감이 지나지 않은 D-15 이하의 할 일.
- 이벤트 포함 범위: schedule-YYYY-MM-DD.md 날짜 기준 지나지 않은 D-30 이하의 이벤트.
- 일정표 필드 구성: 위치, 비고만 작성. registry-schedule 루틴의 '반복' 필드와 일정의 '날짜' 필드는 일정표에 포함하지 않는다.
- 마감 시각이 없는 경우: {YYYY}년 {MM}월 {DD}일 {EEEE(ko)}까지만 작성
- 마감 시각이 정각인 경우: {YYYY}년 {MM}월 {DD}일 {EEEE(ko)} {A(ko)} {hh}시까지만 작성
- 기상 시각: 정각 여부에 관계없이 {A(ko)} {hh}시 {mm}분 전부 작성. 정해지지 않았으면 '미정' 작성
- 위치: 물리적 위치, 도로명주소, URL, 이메일, 연락처, 계좌번호, 파일 경로 등 맥락에 맞는 접근 정보 작성. 없으면 ‘없음’ 작성. 아직 정해지지 않았으면 ‘미정’ 작성
- 섹션(##) 위로 빈 줄 2개. 항목(###) 위로 빈 줄 2개. 그 외 빈 줄 없음.
- 비고: 맥락에 따라 유연하게 작성. 없으면 ‘없음’ 작성.

## Rules


### Principles
- registry-schedule이 유일한 원본이다. schedule-YYYY-MM-DD.md는 원본의 파생물이다.
- 읽기가 쓰기보다 먼저다. 기존 파일과 registry-schedule을 모두 확인한 뒤에 작성한다.
- 일정표는 루틴과 일정의 합집합이다. 출처가 다르더라도 시각순으로 하나의 흐름에 배치한다.
- 기상 시각은 schedule-YYYY-MM-DD.md 생성 시 사용자에게 물어본다.
- D-Day는 반드시 schedule-YYYY-MM-DD.md 날짜를 기준으로 계산한다. 현재 날짜가 아니다.
- 구조가 데이터를 보호한다. 섹션 순서, 헤딩 계층, 필드 구성은 변경하지 않는다.
- 정렬은 기록의 일부다. 새 항목을 추가할 때 올바른 위치에 삽입한다.
- 필드는 삭제하지 않는다. 값이 없어도 필드를 유지한다.
- 불참과 중단은 제외의 근거다. 일정 비고의 불참 정보와 루틴 비고의 중단 기간을 확인하여 해당 항목을 일정표에서 제외한다.
- registry-schedule이 기준이다. schedule-YYYY-MM-DD.md 작성 전에 registry-schedule을 먼저 읽는다.
- 부족한 정보는 추측하지 않고 물어본다. 정보가 불확실하면 사용자에게 확인을 요청한다.
- 기록 전 확인은 생략할 수 없다. 기록 전에 반드시 사용자 확인을 거친다.
- 보고는 기록 및 관리의 마무리 단계다. 업데이트를 완료한 뒤 변경 내용을 사용자에게 반드시 안내한다.
- 포함 범위가 완결성보다 우선한다. 할 일(D-15)과 이벤트(D-30)의 범위 밖 항목은 중요도와 관계없이 제외한다.
- 과거 날짜의 schedule-YYYY-MM-DD.md는 생성하지 않는다.
- schedule-YYYY-MM-DD.md는 매일 생성한다.


### Pitfalls
- 마감 시각이 없는 할 일에 시각 필드를 임의로 채우는 실수. 시각이 없으면 날짜까지만 작성한다.
- 마감 시각이 정각인데 분까지 표기하는 실수. 정각이면 ‘오후 3시’처럼 시까지만 작성한다.
- 항목 삭제 후 빈 줄이 3개 이상 연속되는 실수. 섹션 헤더 위 2개, 항목 헤더 위 2개, 그 외 0개를 지킨다.
- 섹션 내 항목이 전부 제거되었을 때 섹션 헤더까지 삭제하는 실수. 빈 섹션이어도 섹션 헤더는 남긴다.
- 오늘이 아닌 날짜의 schedule-YYYY-MM-DD.md를 작업하는데, 현재 날짜를 D-Day 계산에 사용하는 실수. D-Day는 반드시 schedule-YYYY-MM-DD.md 날짜를 기준으로 계산한다.​​​​​​​​​​​​
- 위치가 아직 정해지지 않은 항목에 ‘없음’을 쓰는 실수. 미확정은 ‘미정’, 접근 정보가 존재하지 않는 것이 ‘없음’이다.
- 마감 시각이 없는 할 일을 같은 날짜의 다른 항목 위에 두는 실수. 시각이 없으면 해당 날짜 맨 아래다.
- 할 일 목록이나 이벤트 섹션의 항목 헤더에 HH:mm을 쓰는 실수. 할 일 목록과 이벤트 섹션의 항목 헤더는 D-Day 정보이고, HH:mm은 일정표 섹션의 항목 헤더에만 쓴다.
- D-Day 당일을 D-0으로 표기하는 실수. 당일은 D-Day다.
- 할 일·이벤트의 날짜를 영문이나 ISO 형식으로 쓰는 실수. 할 일·이벤트의 날짜는 한국어 형식이다.
- 할 일의 필드명을 ‘날짜’로 쓰는 실수. 할 일은 ‘마감’이고, ‘날짜’는 이벤트의 필드명이다.
- 비고나 위치 필드를 빈칸으로 남기는 실수. 값이 없으면 ‘없음’, 미확정이면 ‘미정’을 명시한다.
- presenter-schedule로 registry-schedule를 업데이트하는 실수. registry-schedule는 manager-schedule로 업데이트한다.
- schedule-YYYY-MM-DD.md의 제목 및 내용에 날짜를 잘못 작성하는 실수. 상황과 맥락에 맞는 정확한 날짜를 작성한다.
- 불참 루틴을 일정표에 작성하는 실수. 일정의 비고에 특정 루틴 불참이 작성되어 있으면 그에 해당하는 루틴은 일정표에서 제외한다.
- 일시 중단 중인 루틴을 일정표에 포함시키는 실수. 루틴 비고의 중단 기간에 해당하는 날짜면 해당 루틴을 일정표에서 제외한다.
- 해당 날짜 이전에 마감된 할 일을 할 일 목록에 포함시키는 실수. schedule-YYYY-MM-DD.md 날짜 기준 이미 마감이 지난 항목은 포함하지 않는다.
- 마감 시각이 작성되어 있지 않은 할 일을 00:00 혹은 23:59 마감으로 착각하는 실수. 마감 시각이 작성되어 있지 않은 할 일은 시각을 다루지 않는다.
- registry-schedule를 업데이트하지 않고 schedule-YYYY-MM-DD.md을 작성하는 실수. manager-schedule로 registry-schedule를 먼저 업데이트해야 한다.
- ~/.openclaw/workspace/memory/ 경로에 이미 schedule-YYYY-MM-DD.md 파일이 있는데 읽지 않고 새로 작성하는 실수. 기존 파일을 먼저 읽어야 한다.


### Edge Cases
- 블록이 자정을 넘기는 경우: 예외는 없다. {HH:mm}-{HH:mm} | {블록 이름}으로 헤더를 작성한다. (예시: 22:00-01:00 | 심야 자습)
- 항목이 충돌하는 경우: 허용하지 않는다. 사용자에게 즉시 알린다.
- 기존 schedule-YYYY-MM-DD.md에 사용자가 직접 추가한 항목이 있는 경우: 사용자에게 확인받은 후 manager-schedule로 해당 내용을 registry-schedule에도 반영한다. 단, 기상 시각은 schedule-YYYY-MM-DD.md에서만 다룬다.
- 할 일 마감이 schedule-YYYY-MM-DD.md 날짜 당일 자정(00:00)인 경우: 전 날 23:59 마감으로 변경 후 사용자에게 알린다.
- registry-schedule 업데이트 후 기존 schedule-YYYY-MM-DD.md과 내용이 달라진 경우: registry-schedule 기반으로 업데이트한다.
- 해당 날짜에 루틴도 일정도 없는 경우: 일정표 섹션 헤더는 유지하고 항목 없이 둔다.
- D-15 이내 할 일이 없는 경우: 할 일 목록 섹션 헤더는 유지하고 항목 없이 둔다.
- D-30 이내 이벤트가 없는 경우: 이벤트 섹션 헤더는 유지하고 항목 없이 둔다.
- 할 일의 D-Day가 정확히 D-15인 경우: 포함한다. "D-15 이하"이므로 경계값은 포함 대상이다.
- 이벤트의 D-Day가 정확히 D-30인 경우: 포함한다. "D-30 이하"이므로 경계값은 포함 대상이다.
- 사용자가 먼 미래(30일 이상) 날짜의 schedule을 요청하는 경우: 생성하지 않는다.
- 루틴의 종료일이 해당 날짜 이전인 경우: 해당 루틴을 일정표에서 제외한다.
- registry-schedule에 없는 항목을 schedule-YYYY-MM-DD.md에 직접 추가 요청하는 경우: 사용자에게 확인받은 후 manager-schedule로 registry-schedule에 먼저 반영하고, 그 뒤에 schedule-YYYY-MM-DD.md를 업데이트한다.
- 사용자가 과거 날짜의 schedule을 요청하는 경우: 생성하지 않는다. 사용자에게 과거 날짜임을 안내한다.
- 여러 날에 걸치는 일정이 해당 날짜를 포함하는 경우: 시작일이면 시작 시각부터 23:59까지, 종료일이면 00:00부터 종료 시각까지, 중간일이면 종일로 표기한다. 비고에 전체 기간을 명시한다.
